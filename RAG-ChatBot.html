RAG-ChatBot 代码库详细说明

本文档说明该代码库的整体定位、启动流程、核心请求链路、主要模块职责，以及当前实现中值得注意的工程细节。

整体定位

这个代码库是一个课程资料 RAG 聊天机器人。它把 docs/ 里的课程脚本文本读入，切成小块，存到 ChromaDB 向量库里。用户在网页里提问时，后端让智谱 AI 判断是否需要搜索课程内容；如果需要，就调用本地的课程搜索工具，从向量库取相关片段，再让模型基于片段生成回答。

技术栈大致如下：

  • 后端：FastAPI
  • 前端：原生 HTML、CSS、JavaScript
  • 向量数据库：ChromaDB
  • Embedding 模型：all-MiniLM-L6-v2
  • 生成模型：智谱 glm-4-flash，通过 OpenAI-compatible Chat Completions API 调用
  • 包管理：uv

运行步骤

运行这个代码库时，需要先准备 Python 3.13、uv 和智谱 API Key，然后从项目根目录同步依赖并启动 FastAPI 服务。后端会同时提供 API 和静态前端页面。

  1. 进入项目根目录：
cd d:\Projects\CodexProjects\RAG-Chatbot
  1. 确认 Python 与包管理工具。项目通过 .python-version 指定 3.13，依赖由 uv 管理；如果 uv 不在 PATH 中，可使用 python -m uv。
  2. 安装或同步依赖：
python -m uv sync

该命令会根据 uv.lock 创建 .venv，并安装 FastAPI、Uvicorn、ChromaDB、sentence-transformers 等依赖。首次执行时可能会下载 Python 3.13、依赖包和 embedding 模型相关缓存。

  1. 配置环境变量。复制 .env.example 为 .env，至少填写 ZHIPU_API_KEY：
ZHIPU_API_KEY=your_zhipu_api_key_here
ZHIPU_MODEL=glm-4-flash
  1. 启动项目。Windows 使用 Git Bash 时，可以运行根目录脚本：
./run.sh

PowerShell 中也可以手动进入后端目录启动：

cd backend
..\.venv\Scripts\python.exe -m uvicorn app:app --reload --port 8000
  1. 等待服务完成初始化。启动时 backend/app.py 会创建 RAGSystem，并在 startup_event() 中读取 docs/ 里的课程文本，写入或复用 ChromaDB 向量库。
  2. 打开页面或 API 文档：
  • Web 界面：http://localhost:8000/
  • API 文档：http://localhost:8000/docs
  1. 验证服务是否正常。浏览器访问 http://localhost:8000/api/courses，或在 PowerShell 中执行：
Invoke-WebRequest -Uri http://127.0.0.1:8000/api/courses -UseBasicParsing

正常响应会包含 total_courses 和 course_titles。如果前端页面能打开、课程接口返回 200，说明后端、静态文件托管和课程加载流程已经基本可用。

核心请求链路

  1. 前端 frontend/script.js 调用 POST /api/query。
  2. 后端 backend/app.py 接收 query 和可选的 session_id。
  3. app.py 调用 rag_system.query(...)。
  4. backend/rag_system.py 组织会话历史、工具定义和用户问题。
  5. backend/ai_generator.py 调用智谱模型。
  6. 如果模型判断需要查课程内容，就触发工具调用 search_course_content。
  7. backend/search_tools.py 调用向量库搜索。
  8. backend/vector_store.py 从 ChromaDB 返回相关课程片段。
  9. 模型拿到工具结果后再生成最终回答。
  10. 后端返回 { answer, sources, session_id }。
  11. 前端展示回答，并把 sources 折叠显示为来源。

简化后的数据流：
docs/*.txt -> 文档解析 -> 文本切块 -> ChromaDB -> 用户提问 -> FastAPI -> 模型判断是否搜索 -> 本地搜索工具 -> 向量检索 -> 模型生成回答 -> 前端展示

后端入口：backend/app.py

backend/app.py 负责 Web/API 层。它创建 FastAPI 应用，配置 CORS 和 TrustedHost，初始化全局 rag_system = RAGSystem(config)，并提供两个主要 API：

  • POST /api/query：问答接口。
  • GET /api/courses：获取课程数量和课程标题。

它还把前端静态文件挂载到根路径： 

app.mount("/", StaticFiles(directory="../frontend", html=True), name="static")

因此访问 http://localhost:8000 时，实际返回的是 frontend/index.html。

RAG 总调度：backend/rag_system.py

backend/rag_system.py 是后端核心编排器。初始化时创建这些组件： 

self.document_processor = DocumentProcessor(...)
self.vector_store = VectorStore(...)
self.ai_generator = AIGenerator(...)
self.session_manager = SessionManager(...)
self.tool_manager = ToolManager()
self.search_tool = CourseSearchTool(self.vector_store)

各组件职责如下：

  • DocumentProcessor：解析课程文档并切块。
  • VectorStore：负责向量存储和检索。
  • AIGenerator：负责调用智谱 AI。
  • SessionManager：负责会话历史。
  • ToolManager：负责注册和执行搜索工具。

它的 query() 是问答主流程。这个项目不是后端固定每次先搜索再生成，而是把搜索工具交给模型，让模型自己决定是否调用工具： 

response = self.ai_generator.generate_response(
    query=prompt,
    conversation_history=history,
    tools=self.tool_manager.get_tool_definitions(),
    tool_manager=self.tool_manager
)

这是一种 tool-calling 风格的 RAG。

文档处理：backend/document_processor.py

backend/document_processor.py 负责把课程文本变成结构化课程和 chunk。它假设课程文件格式类似： 

Course Title: ...
Course Link: ...
Course Instructor: ...

Lesson 0: Introduction
Lesson Link: ...
正文...

主要逻辑包括：

  • read_file()：读取 UTF-8 文本。
  • process_course_document()：解析课程标题、链接、讲师、lesson 编号、lesson 标题、lesson 链接。
  • chunk_text()：按句子切块，并保留一定 overlap。

切块配置来自 backend/config.py： 

CHUNK_SIZE = 800
CHUNK_OVERLAP = 100

因此一个 lesson 的正文会被拆成多个适合 embedding 检索的小片段。

数据模型：backend/models.py

backend/models.py 定义了三个 Pydantic 模型：

  • Course：课程标题、课程链接、讲师、lesson 列表。
  • Lesson：lesson 编号、标题、链接。
  • CourseChunk：用于向量库的一段课程文本，以及它属于哪个课程和 lesson。

一个值得注意的点是，Course.lessons: List[Lesson] = [] 使用了可变默认值。Pydantic 通常会处理很多这种情况，但更稳妥的写法是 Field(default_factory=list)。

向量库：backend/vector_store.py

backend/vector_store.py 负责 ChromaDB 存储和检索。它创建了两个 collection： 

self.course_catalog = self._create_collection("course_catalog")
self.course_content = self._create_collection("course_content")

两者职责不同：

  • course_catalog：存课程级别元数据，例如课程标题、讲师、课程链接、lesson 列表。
  • course_content：存真正用于问答检索的课程 chunk。

搜索流程如下：

  1. 如果提供了 course_name，先在 course_catalog 中语义匹配最接近的课程标题。
  2. 根据匹配到的课程标题和可选 lesson_number 构造 ChromaDB filter。
  3. 在 course_content 中做向量搜索。
  4. 返回 SearchResults。

因此它支持“某门课讲了什么”“lesson 5 讲了什么”“哪个课程提到了 RAG”等问题。

搜索工具：backend/search_tools.py

backend/search_tools.py 是模型 tool calling 的桥梁。CourseSearchTool 暴露给模型的工具名是： 

search_course_content

工具参数包括：

  • query：要搜索的内容，必填。
  • course_name：课程名，可选。
  • lesson_number：lesson 编号，可选。

模型看到工具定义后，可以生成类似这样的工具调用： 

{
    "query": "RAG",
    "course_name": "MCP",
    "lesson_number": 5
}

然后 ToolManager.execute_tool() 会真正执行本地 Python 函数，调用 VectorStore.search()。

CourseSearchTool 还会保存 last_sources。这些来源最后会通过 API 返回给前端显示，例如： 

MCP: Build Rich-Context AI Apps with Anthropic - Lesson 5

模型调用：backend/ai_generator.py

backend/ai_generator.py 负责和智谱 API 通信。这里没有使用官方 SDK，而是使用标准库 urllib.request。

调用地址是：

https://open.bigmodel.cn/api/paas/v4/chat/completions

它主要做三件事：

  1. 构造 system prompt，告诉模型什么时候搜索、什么时候直接回答。
  2. 把本地工具定义转换成 Zhipu/OpenAI function tool 格式。
  3. 如果模型返回 tool_calls，就执行工具，然后再发起第二次模型请求生成最终答案。

因此一次课程相关提问通常会发生两次模型请求：第一次让模型判断是否调用搜索工具，第二次让模型基于搜索结果生成答案。

会话管理：backend/session_manager.py

backend/session_manager.py 是一个内存里的简单 session 管理器。它会给新用户创建： 

session_1
session_2
...

每个 session 保存最近几轮对话。配置为 MAX_HISTORY = 2，因为每轮包含 user 和 assistant 两条消息，所以最多保留最近 4 条消息。

需要注意的是，这是内存存储。服务重启后 session 会丢失，也不适合多进程部署时共享状态。

前端

前端由三个文件组成：

  • frontend/index.html：页面结构。
  • frontend/style.css：样式。
  • frontend/script.js：聊天逻辑。

页面左侧显示课程数量和建议问题，右侧是聊天窗口。script.js 中，页面加载后调用 GET /api/courses，用户发送消息时调用 POST /api/query。Assistant 消息使用 marked.parse() 渲染 Markdown，sources 会显示在折叠区域里。

配置

backend/config.py 读取项目根目录中的 .env： 

ZHIPU_API_KEY=...
ZHIPU_MODEL=glm-4-flash

其他关键配置：

EMBEDDING_MODEL = "all-MiniLM-L6-v2"
CHROMA_PATH = "./chroma_db"
MAX_RESULTS = 5

因为服务是从 backend 目录启动的，所以 ./chroma_db 实际会落在 backend/chroma_db。

值得注意的工程细节

  • SessionManager 是内存存储，重启会丢历史。
  • Course.lessons = [] 建议改为 Field(default_factory=list)。
  • document_processor.py 目前主要支持文本格式，虽然代码判断了 .pdf 和 .docx，但读取逻辑仍是普通文本读取。
  • app.py 里定义了 DevStaticFiles，但最终挂载时用的还是普通 StaticFiles。
  • API key 缺失时会在真正请求模型时才报错。
  • 首次启动可能需要下载 sentence-transformers embedding 模型，环境没缓存时会比较慢。

总结

整体来说，这是一个“工具调用式 RAG”项目：不是后端硬编码每次都搜索，而是把课程搜索能力注册成工具，让模型在回答时自主调用。这个设计比固定 pipeline 灵活，但也更依赖 system prompt 和模型的 tool-calling 稳定性。

用户请求处理流程

下面按真实执行顺序跟踪一次用户请求。假设用户在网页输入：Are there any courses that explain what RAG is?

0. 服务启动前置

服务由 run.sh 启动： 

cd backend && uv run uvicorn app:app --reload --port 8000

FastAPI 入口是 backend/app.py。这里会先创建全局对象： 

rag_system = RAGSystem(config)

RAGSystem 初始化时创建文档处理器、向量库、AI 生成器、会话管理器和课程搜索工具。服务启动后，startup_event() 会读取 ../docs，调用 rag_system.add_course_folder(...) 加载课程资料。

1. 前端发起请求

用户点击发送按钮后，触发 frontend/script.js 中的 sendMessage()。它会读取输入框内容，清空输入框并禁用按钮，把用户消息追加到页面，显示 loading 动画，然后发送请求到后端： 

fetch(`${API_URL}/query`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
        query: query,
        session_id: currentSessionId
    })
})

这里 API_URL = '/api'，所以实际请求是 POST /api/query。

2. FastAPI 接收请求

后端由 backend/app.py 的 query_documents() 接收请求： 

@app.post("/api/query", response_model=QueryResponse)
async def query_documents(request: QueryRequest):

请求体会被 Pydantic 解析成 QueryRequest，包含 query 和可选的 session_id。如果前端没有传 session_id，后端会新建一个，然后进入核心处理： 

answer, sources = rag_system.query(request.query, session_id)

3. RAGSystem 组织上下文

rag_system.query() 先把用户问题包装成 prompt： 

prompt = f"""Answer this question about course materials: {query}"""

如果有 session，就从 SessionManager 取历史对话。随后调用 AI 生成器： 

response = self.ai_generator.generate_response(
    query=prompt,
    conversation_history=history,
    tools=self.tool_manager.get_tool_definitions(),
    tool_manager=self.tool_manager
)

这里的关键是：后端不是自己直接搜索，而是把搜索工具定义传给模型，让模型决定是否调用工具。

4. AIGenerator 调用模型

AIGenerator.generate_response() 会构造 system prompt、用户消息、tools，并设置 tool_choice = "auto"。工具定义由 backend/search_tools.py 提供，工具名是 search_course_content。

随后它调用智谱 Chat Completions API： 

https://open.bigmodel.cn/api/paas/v4/chat/completions

5. 模型决定是否调用搜索工具

如果模型认为这是课程相关问题，就会返回 tool_calls。代码判断到工具调用后，进入 _handle_tool_execution()： 

if message.get("tool_calls") and tool_manager:
    return self._handle_tool_execution(message, api_params, tool_manager)

模型可能生成这样的工具调用： 

{
    "name": "search_course_content",
    "arguments": {
        "query": "RAG"
    }
}

6. ToolManager 执行搜索工具

ToolManager.execute_tool() 根据工具名找到 CourseSearchTool，然后执行： 

results = self.store.search(
    query=query,
    course_name=course_name,
    lesson_number=lesson_number
)

7. VectorStore 向量检索

VectorStore.search() 的检索流程是：如果有 course_name，先匹配真实课程标题；再根据课程标题和 lesson_number 构造过滤条件；最后查询 course_content collection： 

results = self.course_content.query(
    query_texts=[query],
    n_results=search_limit,
    where=filter_dict
)

这一步会用 embedding 模型把 query 转成向量，再从 ChromaDB 中找最相近的课程 chunk。

8. 搜索结果格式化

搜索结果回到 CourseSearchTool._format_results()，被格式化成课程标题、lesson 编号和文本片段。同时工具会保存 last_sources，用于前端显示来源。

9. 工具结果回到模型

工具结果不会直接返回用户，而是追加到模型消息里，然后发起第二次模型调用。这次不再传 tools，而是让模型基于检索到的课程片段生成最终自然语言回答。

10. RAGSystem 收尾

回答回到 RAGSystem 后，它取出搜索工具记录的来源，清空来源缓存，并把本轮问答加入 session 历史： 

sources = self.tool_manager.get_last_sources()
self.tool_manager.reset_sources()
self.session_manager.add_exchange(session_id, query, response)

11. FastAPI 返回 JSON

接口包装响应并返回给前端： 

return QueryResponse(
    answer=answer,
    sources=sources,
    session_id=session_id
)

前端收到的 JSON 类似： 

{
  "answer": "课程中关于 RAG 的解释是...",
  "sources": ["某课程标题 - Lesson 3"],
  "session_id": "session_1"
}

12. 前端渲染结果

前端收到响应后保存新的 session_id，删除 loading 消息，然后调用 addMessage(data.answer, 'assistant', data.sources)。addMessage() 会用 marked.parse() 渲染 assistant 的 Markdown；如果有 sources，会追加一个折叠的来源区域，并滚动到底部。

总流程

用户输入
  -> frontend/script.js sendMessage()
  -> POST /api/query
  -> backend/app.py query_documents()
  -> RAGSystem.query()
  -> SessionManager 取历史
  -> AIGenerator.generate_response()
  -> 智谱模型第一次调用
  -> 模型按需返回 tool_calls
  -> ToolManager.execute_tool()
  -> CourseSearchTool.execute()
  -> VectorStore.search()
  -> ChromaDB 返回课程 chunk
  -> CourseSearchTool 格式化结果并记录 sources
  -> 智谱模型第二次调用生成最终回答
  -> RAGSystem 保存会话并返回 answer/sources
  -> FastAPI 返回 JSON
  -> 前端渲染回答和来源

核心设计点是：这个代码库采用的是“模型驱动工具调用”的 RAG。后端提供搜索能力，但是否搜索、搜索什么关键词、是否带课程名或 lesson 编号，由模型根据用户问题决定。
