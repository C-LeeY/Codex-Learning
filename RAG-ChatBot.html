Course Materials RAG System

RAG-Chatbot 项目介绍

一个面向课程材料的检索增强生成应用：把课程文稿切分、向量化、持久化， 再通过网页聊天界面回答用户关于课程内容的问题。

后端 FastAPI、ChromaDB、sentence-transformers、Zhipu/Z.ai GLM
前端 原生 HTML、CSS、JavaScript，无构建工具
语料 docs/ 中的课程文稿作为 RAG corpus
入口 backend/app.py 提供 API 与静态页面服务

项目定位

该项目是一个“课程材料问答”RAG 应用。它让用户不需要手动翻阅课程文稿， 就能直接询问某门课、某节课或某个主题的内容。系统会先检索相关课程片段， 再调用大模型生成简洁、上下文相关的回答。

它不是通用聊天机器人，而是围绕课程资料构建的专用课程助手。回答中会带上 sources，帮助用户知道答案来自哪门课程或哪一节课。

后端模块

  • backend/app.py：FastAPI 入口，定义 API、CORS、启动加载和静态文件服务。
  • backend/rag_system.py：RAG 编排层，连接文档处理、向量库、模型生成和会话管理。
  • backend/document_processor.py：读取课程文稿，解析课程和 lesson 元数据，并切分文本。
  • backend/vector_store.py：使用 ChromaDB 存储课程目录和课程内容向量。
  • backend/ai_generator.py：调用 Zhipu/Z.ai GLM，并处理 function calling。
  • backend/search_tools.py：定义 search_course_content 工具，供模型检索课程内容。
  • backend/session_manager.py：维护内存会话历史。
  • backend/models.py：定义 Course、Lesson、CourseChunk 数据模型。

核心数据流

1. 加载语料启动时读取 docs/ 下的课程文稿。
2. 文档切分解析课程、lesson，并按句子生成 chunk。
3. 向量存储课程元数据和内容分别写入 ChromaDB collection。
4. 用户提问前端调用 POST /api/query。
5. 检索生成模型调用搜索工具，结合结果生成最终回答。

查询时，系统会把问题包装成课程问答 prompt。模型如果判断需要课程材料，会调用 search_course_content，该工具在 ChromaDB 中做语义搜索，并把相关内容返回给模型。

向量检索设计

项目使用两个 ChromaDB collection：

  • course_catalog：存课程标题、讲师、课程链接和 lesson 列表，用于课程名模糊匹配。
  • course_content：存课程内容 chunk，用于实际语义检索。

这种拆分让用户可以输入不完整课程名，例如 “MCP 那门课”，系统先解析最接近的课程， 再在对应课程内容中检索。

API 接口

POST /api/query

用于课程问答，请求包含用户问题和可选 session。

{
  "query": "What does the MCP course say about tools?",
  "session_id": "session_1"
}

响应包含答案、来源和会话 ID。

{
  "answer": "...",
  "sources": ["MCP: Build Rich-Context AI Apps with Anthropic - Lesson 3"],
  "session_id": "session_1"
}

GET /api/courses

返回课程数量和课程标题列表。

{
  "total_courses": 4,
  "course_titles": ["...", "..."]
}

前端功能

  • 通过聊天框发送课程问题。
  • 支持回车发送和 suggested questions。
  • 助手回答支持 Markdown 渲染。
  • 显示课程数量和课程标题。
  • 回答下方可折叠显示 sources。

配置与运行

1. 进入项目目录

cd D:\Projects\CodexProjects\RAG-Chatbot

2. 准备 .env 配置

如果还没有 .env，先从模板复制：

Copy-Item .env.example .env

然后在项目根目录的 .env 中填写智谱 / Z.ai API Key：

ZAI_API_KEY=your_zhipu_api_key_here
ZAI_MODEL=glm-4.7-flash

3. 安装依赖

本机如果没有直接可用的 uv 命令，可以通过 python -m uv 调用：

python -m uv sync --dev

4. 首次缓存 embedding 模型

项目使用 sentence-transformers/all-MiniLM-L6-v2 生成向量，首次运行前需要下载并缓存一次：

.\.venv\Scripts\python.exe -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('all-MiniLM-L6-v2'); print('embedding model cached')"

5. 启动服务

推荐进入 backend/ 后启动，这与项目里的静态文件和 docs/ 路径设计一致：

cd backend
$env:HF_HUB_OFFLINE='1'
$env:TRANSFORMERS_OFFLINE='1'
..\.venv\Scripts\python.exe -m uvicorn app:app --host 127.0.0.1 --port 8000

后续再次运行时，可以直接重复上面的启动命令。

6. 访问应用

http://127.0.0.1:8000
http://127.0.0.1:8000/docs
http://127.0.0.1:8000/api/courses

7. 运行测试

python -m uv run pytest

当前特点与限制

特点

  • RAG 各层职责清晰，便于理解和二次开发。
  • 支持课程名模糊匹配和 lesson 过滤。
  • ChromaDB 持久化向量数据，避免每次启动都重新构建。
  • 前后端结构轻量，适合教学项目和原型验证。

限制

  • 会话历史只存在内存中，服务重启后会丢失。
  • 文档读取逻辑主要面向文本课程文稿。
  • 当前没有课程上传和重新索引 API。
  • CORS 和 TrustedHost 当前放开为 *，更适合本地开发。
  • 测试覆盖较少，主要覆盖共享 Pydantic 模型。

后续改进方向

  1. 增加课程上传、删除和重新索引接口。
  2. 把 session 存到 SQLite 或 Redis。
  3. 支持 PDF 和 DOCX 的真实解析。
  4. 在 sources 中显示课程链接、lesson 链接和命中文本片段。
  5. 增加检索质量评估测试。
  6. 限制生产环境下的 CORS、TrustedHost 和日志内容。

docs/ 中出现的 provider 或 model 名称是课程原始语料内容， 不代表应用本身依赖这些供应商。项目运行时使用的是 Zhipu/Z.ai 配置。

项目调用流程

这一部分专门跟踪“用户从前端提出问题，到后端如何一步步处理并返回答案”的完整路径。 我们沿着真实代码走，不讲抽象概念，直接对应到文件和职责。

总览

整条链路可以概括成一句话： 用户在前端输入问题，前端把问题发给 /api/query， FastAPI 接住后交给 RAGSystem， 然后由模型决定是否调用课程搜索工具， 检索命中后再生成最终回答，最后把答案和来源返回前端显示。

详细步骤

1. 页面初始化

页面加载后，前端脚本会绑定发送按钮、回车事件和推荐问题按钮， 然后初始化欢迎消息，并调用 /api/courses 加载课程数量和课程标题。 这时前端里的 currentSessionId 还是 null。

作用很简单：把聊天环境先搭好，但还没有真正开始问答。

2. 用户在前端发起问题

用户点击发送按钮，或者在输入框按回车，会触发前端的 sendMessage()。这个函数会先取输入内容，去掉首尾空格， 然后立刻把用户消息插入聊天区，再显示一个 loading 气泡。

之后前端调用： 

POST /api/query

请求体包含两个字段：

{
  "query": "用户输入的问题",
  "session_id": currentSessionId
}

如果这是第一次提问，session_id 就是空值； 如果用户已经连续问了几轮，就会带上之前后端返回的 session。

3. FastAPI 接收请求

后端在 backend/app.py 里定义了 /api/query 接口。 它先把请求解析成 QueryRequest，里面只有 query 和可选的 session_id。

处理逻辑是：

  • 如果请求里没有 session，就创建一个新的会话 ID。
  • 调用 rag_system.query(request.query, session_id)。
  • 把结果包装成 answer、sources、session_id 返回前端。

这里 FastAPI 本身不负责检索，也不负责生成回答，它只是把请求安全地转交给 RAG 编排层。

4. RAGSystem 进入问答编排

真正的问答主流程在 backend/rag_system.py 的 query() 方法里。它会先把用户问题包装成课程问答 prompt， 大意是“回答这个关于课程材料的问题”。

接着它会去 SessionManager 里取当前会话的最近几轮对话， 把这些历史上下文一并交给模型生成器。

这里的几个关键对象是：

  • DocumentProcessor：负责课程文稿解析和切分。
  • VectorStore：负责 ChromaDB 向量检索。
  • AIGenerator：负责调用 Zhipu/Z.ai GLM。
  • ToolManager：负责把课程搜索工具暴露给模型。
  • SessionManager：负责保存会话历史。

5. AIGenerator 组织消息并调用 GLM

AIGenerator.generate_response() 会把系统提示词、 当前问题和会话历史组织成 messages。 系统提示词里已经明确约束了模型：

  • 普通问题可以直接回答。
  • 课程相关问题优先搜索再回答。
  • 每次最多做一次搜索。
  • 不要把内部搜索过程讲给用户听。

如果当前传入了工具定义，它还会把 tools 和 tool_choice="auto" 一起传给模型。 这意味着：是否检索，不是后端硬编码决定，而是由模型基于提示词自行判断。

6. 模型决定是否调用搜索工具

第一次调用 Zhipu/Z.ai 后，会出现两种结果：

  • 模型直接返回文字回答，不调用工具。
  • 模型返回 tool_calls，要求执行 search_course_content。

如果没有工具调用，流程就很短：后端直接把模型回答返回。 如果有工具调用，就进入真正的 RAG 检索阶段。

7. ToolManager 执行课程搜索工具

后端暴露给模型的核心工具是： 

search_course_content(query, course_name?, lesson_number?)

这个工具定义在 backend/search_tools.py 中。 模型可以只传查询词，也可以额外指定课程名和 lesson 编号。

工具执行时会调用 VectorStore.search()， 也就是把“自然语言问题”转成向量检索请求。

8. VectorStore 在 ChromaDB 中检索

VectorStore 维护两个 collection：

  • course_catalog：课程标题、讲师、lesson 列表等元数据。
  • course_content：真正参与语义检索的课程内容 chunks。

当模型带了 course_name 进来时，系统会先在 course_catalog 里做一次“课程名解析”。 这一步是为了支持模糊匹配，比如用户只说 “MCP 那门课”， 系统也能匹配到完整课程标题。

然后它根据课程名和 lesson 编号构建过滤条件， 再去 course_content 里做真正的向量检索， 取回命中的文档片段、元数据和距离分数。

9. 搜索结果被格式化并回传给模型

搜索工具不会直接把原始 Chroma 结果交给前端，而是先把结果格式化成模型更容易用的文本。 每一条命中内容前面都会加上类似下面的上下文头： 

[课程标题 - Lesson 3]

同时，工具还会顺手记录一份 sources 列表， 例如“某课程 - Lesson 3”。这份 sources 稍后会被前端显示出来， 所以它不是模型随便写的，而是检索工具真实产生的结果。

10. 模型基于检索结果生成最终回答

当工具执行完之后，AIGenerator 会把工具结果作为 一条 role="tool" 的消息追加回对话，再次调用 Zhipu/Z.ai。

所以这里其实是两段式生成：

  1. 第一次问模型：要不要检索。
  2. 第二次问模型：基于检索结果生成最终答案。

这也是这个项目的核心 RAG 模式。

11. RAGSystem 收尾：sources 与会话历史

当最终回答返回后，RAGSystem 会做三件小但关键的事：

  • 从工具管理器取出本轮检索生成的 sources。
  • 清空工具内部缓存，避免下次串数据。
  • 把“用户问题 + AI 回答”写入 SessionManager。

这意味着后面用户继续追问时，模型可以拿到最近几轮上下文， 但这个上下文只保存在内存里，服务重启后会消失。

12. 前端收到响应并渲染结果

FastAPI 返回的 JSON 里包含： 

{
  "answer": "...",
  "sources": ["课程 A - Lesson 2", "课程 B - Lesson 4"],
  "session_id": "session_1"
}

前端收到后会：

  • 如果当前没有 session，就保存后端返回的 session_id。
  • 移除 loading 气泡。
  • 把 answer 作为 assistant 消息插入聊天区。
  • 如果有 sources，在回答下方渲染一个可折叠来源面板。

助手回答会经过 Markdown 渲染，所以最终看到的是更自然的格式化输出。

你可以这样理解整条链路

这个项目不是“前端直接搜数据库”，也不是“后端每次都强制检索”。 它更像一个由大模型驱动的课程助手：

  1. 前端负责收集问题和展示结果。
  2. FastAPI 负责把请求转交给 RAGSystem。
  3. RAGSystem 负责组织上下文、工具和检索流程。
  4. GLM 负责判断要不要查课程材料，并生成最终答案。
  5. ChromaDB 负责提供真正的课程语义检索能力。

换句话说，用户看到的是一次问答，后端实际走的是 “会话管理 + 工具调用 + 向量检索 + 二次生成”的组合流程。
